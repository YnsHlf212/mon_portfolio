Vue d'ensemble

ObsiLock est un coffre-fort numérique sécurisé permettant aux utilisateurs de déposer, organiser, partager et suivre des fichiers avec chiffrement au repos obligatoire. Le projet est une réalisation professionnelle multi-équipes combinant une API REST (PHP/Slim), un client lourd (JavaFX) pour le dépôt/gestion, et un client web pour la consultation et téléchargement via liens partagés.

Développé en 7 jours, le projet met en avant les bonnes pratiques de sécurité (JWT, Argon2id, AES-256-GCM), le travail en équipe via GitHub, les tests complets, et la conteneurisation Docker. La plateforme gère l'authentification, le versionnage de fichiers, les quotas d'espace, et la journalisation complète des actions.

Fonctionnalités Principales

  • Création de comptes et authentification stateless par JWT
  • Stockage chiffré obligatoire (AES-256-GCM) avec clés enveloppe (RSA-OAEP/X25519)
  • Organisation hiérarchique de dossiers et fichiers
  • Versionnage automatique : remplacer un fichier crée une nouvelle version
  • Liens de partage sécurisés avec expiration ou limite d'utilisations
  • Révocation immédiate de liens par le propriétaire
  • Quotas d'espace par utilisateur avec alertes (80%/100%)
  • Journalisation complète des téléchargements et accès (date, IP, UA)
  • Client JavaFX pour upload/gestion avec barre de progression
  • Client Web public pour téléchargement sécurisé via token

Schéma Base de Données

6 entités principales :

  • users – id, email, pass_hash (Argon2id), quota_total, quota_used, is_admin, created_at
  • folders – Hiérarchie : id, user_id, parent_id, name, created_at
  • files – id, user_id, folder_id, original_name, mime, size, created_at
  • file_versions – Historique : id, file_id, version, stored_name, IV, auth_tag, key_envelope, checksum, created_at
  • shares – Partages : id, user_id, kind (file|folder), target_id, label, expires_at, max_uses, remaining_uses, is_revoked, created_at
  • downloads_log – Journalisation : id, share_id, version_id, downloaded_at, ip, user_agent, success

Points clés :

  • Versionnage : lien vers dernière version par défaut, option pour figer une version
  • Archivage logique des fichiers et utilisateurs
  • Clé par version : chiffrée par clé publique serveur
  • Traçabilité : logs des téléchargements avec IP, UA et succès/échec

Défis Techniques Relevés

Chiffrement Asymétrique & Enveloppe de Clés

Implémentation d'une double couche de chiffrement : AES-256-GCM pour le contenu (symétrique), RSA-OAEP ou X25519 sealed box pour protéger les clés de fichier (asymétrique). Gestion sécurisée des clés privées serveur sans exposer les secrets.

Travail Multi-équipes & GitHub

Coordination de 4 rôles (Back/JavaFX/Web/Ops) sur 7 jours avec OpenAPI source-de-vérité, branching stratégique (main/dev/features), PR + revues de code, Kanban board et daily meetings.

Versionnage & Compatibilité des Liens

Système de versionnage transparent : remplacer un fichier crée une nouvelle version, les liens existants pointent automatiquement vers la dernière version (avec option révision figie). Gestion des compteurs d'utilisation et de l'expiration.

Client Lourd JavaFX avec Upload Progressif

Développement d'une interface JavaFX (FXML) avec gestion complète de l'authentification, arborescence de dossiers (TreeView), tableau de fichiers (TableView), upload multi-part avec barre de progression, et affichage de quota en temps réel.

Contrat d'API (OpenAPI)

Authentification :

  • POST /auth/register {email, password} → 201
  • POST /auth/login {email, password} → 200 {jwt}

Dossiers & fichiers :

  • GET/POST/DELETE /folders
  • GET /files?folder={id}
  • POST /files (multipart) → crée version 1
  • POST /files/{id}/versions (multipart) → nouvelle version
  • GET /files/{id}/download (auth)

Partages :

  • POST /shares {kind, target_id, expires_at|max_uses}
  • GET /shares (listes + stats)
  • POST /shares/{id}/revoke
  • GET /s/{token} (public)
